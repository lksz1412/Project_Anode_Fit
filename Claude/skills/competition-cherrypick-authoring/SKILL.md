# 경쟁·체리픽 다중에이전트 저작법 (N+N+1+1)

## 0. 언제 쓰나 (트리거)
- 회사 배포·교육·근거가 되는 **고가치 레퍼런스 산출물**(교과서급 문건·핵심 코드·정본 데이터).
- 결함이 **기계 검증으로 안 걸리는** 종류(물리 서술·유도 다리·부호·그림·아키텍처 선택).
- 사용자가 토큰 비용을 수용(이 방법은 N=9 기준 작성 ~20 + 검수 ~25 에이전트). **경량 작업엔 쓰지 말 것.**
- ★사용자 명시 opt-in 또는 본 스킬 호출 시에만. 임의 발동 X.

## 1. 골격 (N+N+1+1)
```
① N 독립 작성  → 검토1 → ② N 보완(방향성-만) → 검토2 → ③ 체리픽 통합(1) → ④ 재검수→최종(1)
```
- 기본 N=9(모델 3종 × 3). 규모는 작업 난도에 맞춰(빠른 확인 N=3, 정본 N=9). 후보는 많을수록 체리픽 質↑(규모의 경제).
- ★**Fable 가용 시 N=10**: 작업 시점에 Fable 사용이 가능하면 **Fable ×1 을 추가**(3 Sonnet + 3 Opus + 3 Codex + 1 Fable). 검토·점수 집계에서 **Fable 산출물/검토건에 가중치 3** 을 부여해 모델 계열별 동등 가중을 맞춘다(Sonnet 3×1 = Opus 3×1 = Codex 3×1 = Fable 1×3). Fable 불가 시 기존 N=9 그대로.
- **전부 Agent 도구**(Workflow·권한팝업 도구 금지). **단계마다 사용자 팝업 0**(GO 후 무중단). **푸쉬는 사용자 정책 따름**(기본: 커밋만, 푸쉬 X).

## 2. Phase 0 — 공유 골격(spine) 먼저
1. **정본/원천 정독**(master): 산출 목표·정합 기준(코드·사양). 곡선/구조의 *진행 순서*를 플로우차트/골격으로 고정.
2. **AUTHOR_BRIEF**(공유 사양 1장): 역할·경계(작업 sub·범위밖 금지·허위 attribution 금지)·목표·골격·형식 규범·★검수 렌즈 체크리스트·산출 경로·자체검수 규약.
3. **KNOWN_DEFECTS.md**(누적): 발견되는 상속/系統 결함을 적어 두고 매 검토·체리픽서 전수 점검(컴팩션-안전·전파 추적).
4. 동일 브리프로 공정 경쟁 — 다양성은 모델·자율에서 나오게(인위 다양화 X).

## 3. Phase ① — N 독립 작성
- 모델 배분 예: Sonnet ×3·Opus ×3·Codex ×3(★Fable 가용 시 + Fable ×1, 가중치 3 — §1). **상호 무통신**(독립이 다양성의 핵심).
- 각 에이전트: 골격 따라 절별/단위별 루프 + 자체 N회 검수(토큰 큰 작업이면 사용자 합의 하에). 산출 = 완결물 + NOTE(Read Coverage·자체검수 로그).
- ★**Codex 는 단계 구동 + 지속 관찰**: 한 번에 다 시키고 방치 X. `codex-companion.mjs status [job-id]`·`result [job-id]` 로 폴링. one-shot 은 깜깜이·중단 위험(실측 교훈).
- master 검증(에이전트 자기보고 신뢰 X): 빌드·정합·게이트를 **직접 재실행**. → 커밋 #1.

## 4. Phase 검토1·검토2 — 교차 검수 (2회·다른 청크)
- 산출당 검수 에이전트 1마리(리뷰 전용·파일만 생성). 분량 크면 ~500줄 청크 분할(통째 위임 = "100% 정독" 과장).
- ★**렌즈를 산출 종류에 맞게**(refute mandate·가장 약한 1곳 필수·빈 통과 금지):
  - 공통: 정합(정본 1:1)·완결성(orphan 0)·시각/그림.
  - 코드: 부호/시그너처/호출체인/silent path.
  - 문건: **G-follow(앞 식만으로 따라가나)·G-usable(이 문건만으로 재현되나)·G-derive(각 식이 단계로 *유도*되나, "대입하면 [박스]" 점프 0)** — 정확성만 보면 통과되는 결함을 잡는 1급 렌즈.
- 점수화 + KNOWN_DEFECTS 보유표 + 체리픽 베이스 식별.
- **검토2 는 검토1 과 *다른 청크 스킴*** + ★**보완이 새 결함을 유입했는지**(회귀) 1급 점검 — "고치다 다시 틀림"이 흔하다(실측: 보완 회귀 다발).
- ★**변형 — 검수 N종 유니온 + 10차 재검**(사용자 확정 운용, 완성도 최우선 시): 검수도 작성과 같은 N종(모델 분산·무통신)으로 돌려 **단점은 전부 합집합(union)** 으로 모으고(장점은 체리픽), 별도 세션의 **10차 재검**이 union 각 항목을 진짜결함/강등/오적발로 필터한다(검수자 과민·산술오류·물리동치 미인지가 실측 다발 — 오적발을 그대로 반영하면 멀쩡한 물리를 훼손). 기록(SPEC·계획서·FIX_LIST)을 근거로 대조.
  - ★**교차합의 triage(2026-07 v1.0.15 정련)**: union 항목을 **발견 검수자 수**로 1차 분류한다 — **≥2종이 공통 지적 = 진짜 결함일 가능성 높음(채택 후보)**, **1종 단독 지적 = master 정밀 재검**(원문 재정독·재유도)으로 진위 확정 후에만 채택한다(단독일수록 오적발 위험 大 — 단독 지적을 그대로 반영하면 멀쩡한 물리 훼손). 즉 **교차합의 = 오적발 필터 1차 신호, master 재검 = 2차 신호**. 재검·판정 담당은 §7 규칙(기본 Opus·Fable 가용 시 Fable).

### 4.1 응용 — 상위 모델 유산 산출물의 검증-전용 모드 (수정 X·보고만)
이미 **상위 모델(예: Fable)이 작성한 완성도 높은 산출물**을 다룰 때: **기본 무수정 보존**이 골조이고, 논리 결함만 점검한다. 작성 단계(①②③) 없이 **검수 union 단계만** 돌린다.
- **모델 선택 = 고등 검수 우선**: 문건 물리·화학 논리 검토엔 Sonnet 을 빼고 **Codex + Opus** 로(사용자 운용 예 **N=6 = Codex 3 + Opus 3**). 코드·기계 정합 스윕은 Sonnet 병용 가능.
- **triage = 교차합의(위 4)**: ≥2종 공통 = 진짜 문제로 인식, 1종 단독 = master 정밀 재검으로 진위 확정.
- **산출 = 수정이 아니라 소유자에게 보고**(4-tier: 확정 결함 / 단독-확정 / 오적발-기각 / 근거미발견). **수정 여부는 소유자(사용자)가 결정** — 검증자가 상위 모델 물리를 임의 개수(改修)하지 않는다.
- Fable 불가 시 통합·재검·최종 판정 = Opus(§7).

## 5. Phase ② — 방향성-만 보완
- 각 *원* 모델에 재지시: **구체 코드/식 제공 금지 — 방향성만**(약점의 *방향*만, 해법은 자율). REVIEW 원문 비열람 권장(베끼기 방지·자율 보존).
- 산출 = 원본 비파괴 별도본(`*b`). 보완 후 ★**새 회귀 self-test**(직전 교훈). → 커밋 #2.

## 6. Phase ③ — 체리픽 통합 (master 직접; ★Fable 가용 시 Fable 이 수행)
- 베이스 = 최고점·최소결함 1본. **graft = 각 산출의 검증된 강점**(절·유도·그림·구조)만.
- ★**이식 금지 목록**: 검토2가 잡은 보완 회귀·botch된 수정은 절대 채택 X(명문화).
- **단계적 참고문헌**: 초기 N 에는 핵심 원천만(다양성↑), 체리픽 단계에서 교정·정합용 추가 원천 합류(정확성↑) — 발산과 수렴을 분리.
- 그림도 경쟁 대상: 후보 풀(기존 + 신규)에서 이해를 가장 돕는 것 선택. → 커밋 #3.

## 7. Phase ④ — adversarial 재검수 → 최종
- 체리픽본을 **한 번에 OK 금지** — 신선한 적대 검수(렌즈 분담)로 잔여 결함·이음새 적발(빈 통과 거부).
- master 가 보완 → 최종본 + **정식 N회 수렴 검수**(연속 2R 0결함·매 라운드 청크·렌즈 전환). 누적 검토(검토1·2·재검수·최종)로 N회 초과 커버를 정직 회계.
- ★**10번째(체리픽 통합·최종본) 작성자 = 기본 Opus 이나, 작업 시점 Fable 가용이면 Opus 대신 Fable 을 사용한다**(§6 통합·본 §7 최종 마감 공통).
- → 커밋 #4. + 종합 RESULT(11항목)·ledger(12-col)·메타평가.

## 8. 운용 규율 (하드)
- result/ledger 를 **매 Phase 디스크 저장**(컴팩션 복구지점·sub 결과 대화에만 남기지 말 것).
- **원본 불가침**(접미 사본). 결과 박스/식별자/부호 등 검증된 핵심은 체리픽서 보존(확장만).
- 커밋 시점 = ①N초본 / ②N보완 / ③통합 / ④최종(4회). 푸쉬는 사용자 정책.
- ★Codex 단계구동+관찰(§3). 529/토큰초과 등 일시 오류는 재가동(과부하 중 연속 재시도 X — 시간 두고).

## 9. 왜 효과 있나 (실측 근거)
- **다양성이 系統/상속 결함을 표면화**: 단일 저자면 출하했을 오류를 *대조*가 드러냄(예: 코드 충전꼬리 부호 4/9·문건 D-PEAK 상속 8/9).
- **2차 검토가 보완 회귀 차단**: 고치다 다시 틀린 것을 잡아 통합서 배제.
- **G-derive/G-follow/G-usable 렌즈**가 "맞지만 못 따라가고 못 쓰는" 결함을 잡음(정확성 렌즈만으론 통과).
- **체리픽 최종 > 임의 단일본**: 최고 베이스 + 검증된 graft.
- 한계: 토큰 비용 大·Codex 는 분량/유도 약함(역할 한정)·중첩 self-review 에이전트는 통지 복잡(파일로 수렴).
